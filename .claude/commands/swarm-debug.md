# Swarm 디버깅

경쟁 가설 디버깅으로 버그 근본 원인을 파악합니다.

## 절차

1. **버그 현상 수집**: 재현 조건, 에러 메시지, 스택 트레이스, 기대 동작 vs 실제 동작

2. **3-5명의 디버깅 팀 생성** (에러 유형에 따라 에이전트 선택):

   | 에러 유형 | 추천 에이전트 | 역할 |
   |-----------|--------------|------|
   | 빌드/타입 에러 | `build-error-resolver` | TypeScript, 번들러 에러 추적 |
   | 로직/런타임 에러 | `code-reviewer` | 코드 흐름, 상태 관리 분석 |
   | 아키텍처/설계 문제 | `architect` | 구조적 원인, 의존성 문제 |
   | 보안 취약점 | `security-reviewer` | 인증/인가, 입력 검증 문제 |
   | 종합 조율 | `planner` | 가설 정리, 증거 종합 |

   - 버그 성격에 맞는 3-5명 선택 (모두 사용할 필요 없음)
   - 각 팀원에게 서로 다른 가설 할당
   - 팀원들이 서로의 가설을 반박하도록 지시

3. **과학적 토론 방식**:
   - 각 팀원이 자신의 가설을 증거와 함께 검증
   - 코드를 읽고, 로그를 분석하고, 재현 시도
   - 다른 팀원의 가설을 적극적으로 반박
   - 증거 기반 합의 도출

4. **결과 종합**: 근본 원인 + 수정 방안 제시

## 출력 형식

```
## 디버깅 결과

### 근본 원인
(합의된 원인 설명)

### 증거
- (각 팀원이 발견한 핵심 증거)

### 기각된 가설
- 가설 A: (기각 이유)

### 수정 방안
1. (우선순위 순)
```

## 참고
- 단순한 버그는 단일 세션이 더 효율적
- 재현이 어렵거나 원인이 불명확한 버그에 적합
- 각 팀원은 코드만 읽고, 수정은 합의 후 리드가 수행

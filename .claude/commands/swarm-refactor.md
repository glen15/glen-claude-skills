# Swarm 리팩토링

대규모 리팩토링의 모듈별 병렬 작업을 수행합니다.

## 절차

1. **리팩토링 범위 분석 및 모듈 분리**:
   - 영향 범위 파악 (파일, 디렉토리, 의존성)
   - 독립적으로 작업 가능한 모듈 단위로 분리

2. **모듈별 팀 구성**:
   - 각 팀원에게 독립 모듈/디렉토리 할당
   - 파일 소유권 명확히 분리 (충돌 방지)

3. **Plan Approval 필수**:
   - 각 팀원이 리팩토링 계획 제출
   - 리드가 인터페이스 호환성 검증 후 승인
   - 공유 인터페이스 변경은 리드가 먼저 수행

4. **리팩토링 원칙**:
   - 기존 테스트 먼저 통과 확인
   - 구조 변경과 기능 변경 분리 (Tidy First)
   - 각 모듈 완료 후 즉시 테스트

5. **통합 테스트로 전체 호환성 검증**

## 출력 형식

```
## 리팩토링 결과

### 변경 요약
- 모듈 A: (변경 내용)
- 모듈 B: (변경 내용)

### 테스트 결과
- 단위 테스트: (통과/실패)
- 통합 테스트: (통과/실패)

### 후속 작업
- (있다면)
```

## 참고
- 단일 파일/모듈 리팩토링은 단일 세션이 더 효율적
- 두 팀원이 같은 파일을 편집하면 안 됨
- 리팩토링 커밋과 기능 커밋을 반드시 분리
